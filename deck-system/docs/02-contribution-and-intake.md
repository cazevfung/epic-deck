# 02 Contribution And Intake

## Purpose

This document explains how content should enter the deck system.

Its job is to make incoming material usable before page design starts.

Use it for:

- contributor instructions,
- intake normalization,
- source-slide migration prep,
- page brief readiness checks.

## Core Rule

Departments contribute `content`, not final slide design.

If contributors submit polished pages with inconsistent visual systems, the consolidation workload increases sharply and the deck loses coherence.

## Default Submission Format

Use [`../intake/page-brief-template.md`](../intake/page-brief-template.md) as the default page brief.

Every requested page should produce one normalized brief.

## What Intake Must Solve

Different contributors will submit material with different problems:

- multiple arguments on one slide,
- titles that are really paragraphs,
- proof mixed with decoration,
- unlabeled screenshots,
- repeated data with unclear priority,
- logos and examples that feel important but are not explicitly ranked,
- pages that only exist because of an old layout, not because they are the right message unit.

The intake process should convert that material into one clear page request.

## What Contributors Should Submit

Each requested page should come with a brief that includes:

- page role,
- one-sentence page goal,
- one-sentence takeaway,
- primary claim,
- must-keep proof,
- supporting proof,
- key numbers and sources,
- required visual assets,
- suggested page type,
- likely fit risks,
- priority level.

## Required Intake Fields

Every normalized brief must define:

- source page or source material reference,
- page role in the deck,
- one-sentence page goal,
- one-sentence audience takeaway,
- primary claim,
- must-keep proof,
- supporting proof,
- required metrics and sources,
- required named examples,
- required assets and ownership,
- suggested template family if known,
- likely fit risks,
- priority level.

If those fields are not clear, the page is not ready for design.

## Page Role

The brief should state the page's job in the deck sequence.

Typical page roles:

- context,
- market proof,
- capability proof,
- strategic thesis,
- plan,
- timeline,
- budget,
- operations,
- closing.

Page role matters because two pages may use similar content but require different framing depending on where they sit in the story.

## Message Rules

Each brief must define:

- what this page is trying to make the audience believe,
- what the audience should remember after seeing it,
- what evidence is doing the convincing.

If the page has two unrelated takeaways, split the brief before layout begins.

## Proof Priority Rules

Every proof item should be classified before design:

- `Must Keep`: cannot disappear or become unreadable,
- `Support`: should remain if the page can carry it cleanly,
- `Replaceable`: may be swapped for equivalent proof,
- `Move`: belongs on another page, caption, appendix, or speaker note,
- `Drop`: intentionally removed as redundant.

This prevents layout work from silently deciding content priority.

## Template Suggestion Rules

Contributors may suggest a template, but the suggestion is provisional.

The deck editor should confirm whether the page is really:

- `T03` one judgment,
- `T04` judgment plus proof,
- `T05` three parallel drivers,
- `T06` one example proving a broader point,
- `T07` structured comparison,
- `T08` roadmap or phased sequence,
- `T09` resource allocation,
- `T10` closing or hero statement.

If no standard family fits cleanly, mark the brief as an exception candidate instead of forcing a bad template choice.

## Fit Risk Flags

Each intake brief should flag risks early.

Common risk flags:

- too many competing messages,
- too many proof modules,
- unclear main visual,
- weak or missing source,
- logo density risk,
- unreadable screenshot risk,
- likely split candidate,
- likely named-recipe candidate,
- compliance-sensitive claim,
- placeholder-heavy page.

Risk flags are not failures.

They are signals that the page needs stronger editorial handling before visual polish.

## Submission Workflow

1. Define the page goal.
2. Write the one-sentence takeaway.
3. Separate must-keep proof from supporting proof.
4. Attach only the data and assets needed to support that takeaway.
5. Suggest a page type if clear.
6. Flag likely fit risks early.
7. Mark the page as `Must Have`, `Should Have`, or `Optional`.

## Existing Page Migration Workflow

When redesigning an existing page, do not start by making a cleaner layout.

Start by preserving the source inventory.

### Step 1: Inventory The Source Page

Record what exists before changing the page:

- title and takeaway,
- all metrics and labels,
- all named examples and channels,
- all images, logos, screenshots, charts, and thumbnails,
- all captions, source notes, and caveats,
- the page's role in the deck sequence.

### Step 2: Classify Each Item

Every source item should be tagged as one of:

- `Keep`: must remain visible,
- `Merge`: can be combined with nearby copy while preserving meaning,
- `Replace`: can be swapped for equivalent evidence,
- `Move`: can move to caption, card, appendix, or another page,
- `Remove`: truly redundant and explicitly recorded.

### Step 3: Choose Layout After Inventory

Only choose the template after the source inventory is clear.

If the page cannot fit without losing `Keep` items, the layout is wrong.

Use a different template, create a compact proof module, or split the page.

### Step 4: Verify Against The Source

Before marking the page complete, compare old and new:

- no required metric disappeared,
- no required visual proof disappeared,
- no named example was generalized away,
- no caveat or source-sensitive note was hidden,
- any `Remove` decision is intentional and documented.

## Content Writing Rules

- one page should not carry multiple independent arguments,
- every page should begin from a conclusion, not a raw information dump,
- supporting bullets should be short and non-overlapping,
- long background paragraphs should be moved to appendix or source notes.

## Text Structure Rules

Contributors should write content that can fit the standard page text geometry.

- page titles should be short,
- one-sentence takeaway should stay truly one sentence,
- support copy should be scannable rather than essay-like,
- if a point needs a long paragraph, it likely needs a different page structure.

Contributors should not assume that a page can absorb unlimited text simply because there is visual space.

## Header Content Rules

For standard pages, contributors should expect:

- one eyebrow or chapter cue,
- one page title,
- optional short subhead only if necessary.

Do not submit:

- multiple headline candidates for one page,
- title plus another title-sized sentence,
- long subtitles that function as body copy.

## Data Rules

Every key number should include:

- metric definition,
- time scope,
- geography or platform scope if relevant,
- source.

If the source is not final, mark it clearly as `to be confirmed`.

## Asset Rules

For every asset, contributors should provide:

- file name,
- source or owner,
- intended use,
- whether the asset is final or placeholder.

See [`../intake/asset-naming.md`](../intake/asset-naming.md) for naming guidance.

## What Not To Submit

- long raw notes pasted from meetings,
- full paragraphs without a takeaway,
- unlabelled screenshots,
- data without source,
- multiple page ideas bundled into one page brief,
- heavily styled pages that ignore the shared system.

## Ready-For-Design Standard

A page brief is ready for production when:

- the takeaway is clear,
- the primary claim is singular,
- the page can map to a standard template or is explicitly escalated,
- must-keep proof is identified,
- the supporting material is sufficient but not excessive,
- data and asset ownership are clear,
- the fit risks are visible.

## Escalation Rule

Escalate before production if:

- the page needs a new template,
- the page depends on missing data,
- the page mixes too many objectives,
- the content owner cannot define a clear takeaway,
- the page only works as a dense exception but no recipe exists,
- the page likely needs to split into two or more pages.
