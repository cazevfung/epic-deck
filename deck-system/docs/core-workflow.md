# 01 Core Workflow

## Purpose

This is the main operating workflow for rebuild work.

Use it together with:

- [`style-bible.html`](style-bible.html)
- [`canon-first.md`](canon-first.md)
- [`rebuild-guideline.md`](rebuild-guideline.md)
- [`visual-system.md`](visual-system.md)
- [`quality-checklist.md`](quality-checklist.md)
- [`review-gates.md`](review-gates.md)

## Core Rule

Do not redesign the whole deck in one pass.

Work in phases and keep each phase explicit.

## Phase Order

1. Normalize the input.
2. Audit the slide set.
3. Find the closest bible family.
4. Plan rebuild waves.
5. Rebuild slides.
6. Review quality.
7. Extract patterns only if they are real.

## Phase 1: Normalize The Input

Turn each slide into a clear brief:

- page job,
- one-sentence takeaway,
- must-keep proof,
- fit risks,
- split risk if overloaded.

Use [`../intake/page-brief-template.md`](../intake/page-brief-template.md) when needed.

Stop if the slide contains multiple separate arguments or missing proof.

## Phase 2: Audit The Slide Set

Before redesign, note:

- which pages are already close to canon,
- which pages are overloaded,
- which pages feel generic,
- which pages can be grouped into a rebuild wave.

## Phase 3: Find The Closest Bible Family

For each slide, identify the nearest family in [`style-bible.html`](style-bible.html).

Examples:

- dark divider,
- light divider,
- hero statement,
- comparison proof,
- image-led proof,
- proof wall,
- showcase page.

Do not start from a generic card or split layout if a clearer family already exists.

## Phase 4: Plan Rebuild Waves

Group pages into:

- `Wave 1`: closest to canon and lowest risk,
- `Wave 2`: moderate adaptation,
- `Wave 3`: overloaded or exception-heavy pages.

## Phase 5: Rebuild Slides

Build in this order:

1. stage and atmosphere,
2. headline hierarchy,
3. dominant proof,
4. support proof,
5. micro labels and page number.

When content does not fit, respond in this order:

1. clarify the message,
2. re-rank proof,
3. cut duplication,
4. simplify modules,
5. split the page.

Do not solve fit with hidden overflow or blanket shrinkage.

## Phase 6: Review Quality

Check the page against:

- [`quality-checklist.md`](quality-checklist.md)
- [`review-gates.md`](review-gates.md)

The page should pass all three:

- source preservation,
- fit,
- bible similarity.

## Phase 7: Extract Patterns Carefully

Only promote a pattern if:

- the rebuilt page is already strong,
- the pattern clearly belongs to the bible family,
- the pattern is likely to repeat.

Use [`pattern-extraction-policy.md`](pattern-extraction-policy.md).

## Success Standard

This workflow is working if:

- rebuilds look closer to the bible,
- pages stop collapsing into generic layouts,
- proof remains visible,
- fit stays disciplined,
- reusable patterns are extracted after success, not before it.
